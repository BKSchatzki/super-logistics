# v3.1.3 Release Notes: POD Streaming And Invite Observability

## Overview

Version 3.1.3 is a follow-up release focused on two issues found after the `pod_path` schema hotfix:

- large carrier POD files could hang the browser during View/Print because the frontend was trying to render a very large base64 string inside an iframe
- newly created users could appear to be created successfully even when invite-email delivery could not be confirmed

This release keeps WordPress uploads as the storage layer for POD files and improves delivery and observability without changing the underlying POD storage model.

## 1. POD Delivery Refactor

### Invite Problem

The POD view route returned a JSON payload containing a base64-encoded PDF string. That worked for smaller documents, but large or multi-page PODs could cause the browser to stall while decoding and injecting a massive `data:` URL into the iframe.

### Invite Hardening

`TransactionController::printPOD()` now streams raw PDF bytes back to the browser instead of returning base64 JSON.

Frontend behavior changed accordingly:

- the receiver details view now requests POD output as a blob response
- the shared `print()` helper creates a blob URL and opens that in the new window iframe
- other print flows keep their existing base64 behavior

### Why This Helps

- avoids JSON/base64 payload inflation for large PDFs
- avoids browser hangs caused by giant `data:` URLs
- preserves the existing route authorization and transaction-level access checks
- keeps `pod_path` stored as the normal WordPress uploads URL

## 2. Invite Email Observability

### Problem

User creation delegated invite delivery to WordPress via `wp_send_new_user_notifications($userId)`, but the repo provided no visibility into whether the invite-mail step actually succeeded. A new user could appear created successfully even if delivery failed or could not be confirmed.

### Fix

The user creation flow now:

- ensures an `sl_user_status` row exists immediately for the new user
- logs before and after the invite notification attempt
- listens for `wp_mail_failed` and `wp_mail_succeeded` during the invite attempt
- includes a warning payload in the API response when invite delivery fails or cannot be confirmed
- allows the frontend to surface that warning instead of reporting silent success

This does not replace WordPress core invite handling. It improves observability and reduces silent failure from the plugin side.

## 3. Scope Boundary

This release improves the repo-side behavior only.

The following items still require live-environment inspection when access is available:

- active plugin/theme hooks affecting `wp_new_user_notification*` or `wp_mail`
- cPanel mail routing / SMTP configuration
- real invite delivery behavior on `tstg.bigtbx.com`
- live verification of the failing large POD record after deployment

## Validation

Local validation completed for the code changes included in this release:

- PHP syntax checks for updated controllers
- targeted `vitest` coverage for the updated frontend `print()` helper
- production frontend build
