# v3.1.1 Release Notes: Carrier POD System and Fixes

## Overview

Version 3.1.1 addresses client feedback from D. Jones's post-delivery UAT. Tracking export and manifest pagination were confirmed working. The POD feature was reworked from scratch to match the client's actual requirement: uploading and managing carrier-provided proof-of-delivery documents, not generating branded receipts from freight photos.

## Changes

### 1. Carrier POD Upload System (main deliverable)

**What changed:** The previous POD implementation generated a thinkSTG-branded PDF embedding the freight photo (`image_path`). Jones clarified that what's needed is the ability to attach a carrier's own POD document (PDF or JPEG) to a receiver, separate from the freight photo.

**Implementation:**

- **Database:** Added `pod_path VARCHAR(255) NULL` to `sl_transactions` alongside `image_path`. Added to model `$fillable` and transformer output.
- **Backend endpoints:**
  - `POST receiving/pod/upload` (`isInternal`) -- Upload a carrier POD (PDF, JPEG, PNG). Replaces previous POD if one exists.
  - `POST receiving/pod/delete` (`isInternal`) -- Delete the attached carrier POD file and clear `pod_path`.
  - `POST receiving/pod` (`isLoggedIn`) -- View/print the carrier POD. Returns the original PDF directly (pass-through) or wraps images in a minimal single-page PDF. Includes per-transaction authorization so client users can only access their own client's transactions.
- **Upload handler:** `Controller::handleImageUpload()` now accepts an optional `$mimes` parameter. POD uploads pass an extended MIME list that includes `application/pdf`.
- **`PODDocGenerator`:** Rewritten. For PDF files, reads and returns bytes directly. For images, wraps in a minimal TCPDF page. No more branded layout.
- **Frontend (`ReceiverDetails.vue`):** New "Carrier POD" panel with:
  - Status indicator (attached filename or "No carrier POD attached")
  - View/Print POD button (all logged-in users when POD exists)
  - Upload/Replace POD button (internal users only, file picker for .pdf/.jpg/.jpeg/.png)
  - Delete POD with inline confirmation (internal users only)

**Access model per Jones's email:**

| Action     | Internal | Client                          |
| ---------- | -------- | ------------------------------- |
| Upload     | Yes      | No                              |
| View/Print | Yes      | Yes (scoped to own client/show) |
| Delete     | Yes      | No                              |

### 2. printPOD Authorization Hardening

The `printPOD` endpoint permission was changed from `isInternal` to `isLoggedIn` to allow client access. Added `authorizeTransactionAccess()` to enforce that client users can only view PODs for transactions belonging to their assigned client (and for `client_employee`, their assigned shows). The method follows a deny-by-default pattern: internal users are explicitly allowed, client users are explicitly scoped, and all other user types are denied.

### 3. "Seal No: undefined" Fix

**Root cause:** The trailer manifest form's `seal_no` field starts as JS `undefined`. `FormData.append(key, undefined)` serializes it as the string `"undefined"`, which the PHP generator prints verbatim.

**Fix:** Two layers:

- `RequestUtility.createFormData()` now skips `undefined`/`null` values instead of appending them.
- `TrailerManifestGenerator` defensively treats `$sealNo === 'undefined'` as empty.

### 4. Intra-row Overflow Hardening

Both `PalletManifestGenerator::writeCellsWithMultiline()` and `TrailerManifestGenerator::printRowMultiline()` now clamp `$maxHeight` to the usable page height after any page-break-and-header-redraw. This prevents TCPDF auto-break from firing mid-row when a single row is taller than the full page, degrading gracefully (content truncates) rather than breaking layout.

### 5. Gruntfile Packaging Fix

The gruntfile previously only copied `**/*.php` files, which missed non-PHP vendor assets (TCPDF fonts, composer autoload JSON, etc.) and metadata files (`composer.json`, `package.json`). Updated to copy `src/`, `vendor/`, `view/dist/`, and metadata files explicitly. Output zip is now versioned (`super-logistics-v{version}.zip`) and wraps contents in a `super-logistics/` directory.
