# v3.1.4 Readiness: Operational SOP For Invite Deliverability

## Overview

This note captures the current working conclusion for the next readiness pass after `v3.1.3`.

- POD rendering is considered fixed in `v3.1.3`
- the remaining invite issue is currently treated as an operational deliverability problem first, not a primary code bug

This is a readiness/SOP document, not a claim that the repo itself has already shipped a `v3.1.4` code release.

## 1. Resolved: POD Rendering

Status: fixed and verified in `v3.1.3`.

Large carrier POD files are now streamed as raw PDF bytes and opened as blob URLs on the frontend, avoiding the browser hang caused by large base64 payloads embedded into an iframe.

## 2. Current Invite Issue: Delivery Black-Holing

### Symptom

New users such as Robert Scott and Marco Parrotto are created in WordPress but do not receive their invite/reset email.

### Current Working Findings

- Track Delivery reportedly shows the local server accepting the messages
- outbound mail is being sent from `hststg@hola8.donteinferno.com`
- that sender identity does not align with the `tstg.bigtbx.com` site identity
- corporate filtering at Show Strategy is the leading operational explanation for why the messages disappear after acceptance

### Operational Interpretation

If the mail server is technically accepting the message but downstream corporate filters are discarding it because the sender identity is unauthenticated or misaligned, that is an environment and deliverability problem, not something additional plugin code is likely to fix on its own.

## 3. Role Observation

The observed `Role: None` state in the users list should be treated as a follow-up symptom to keep an eye on during verification, not as proof by itself of the root cause.

Current working interpretation:

- user creation is happening
- the broader invite/onboarding workflow is not reaching a clean completion state
- mail delivery or login-related plugin behavior remains the primary operational suspect

If the role issue persists after SMTP and sender alignment are corrected, it should be re-investigated separately.

## 4. Operational SOP For thinkSTG

The recommended next step is environment configuration rather than further repo-side changes.

### Step A: Authenticated SMTP

Install and configure an SMTP plugin such as WP Mail SMTP so WordPress mail is sent through an authenticated mailbox aligned with the site environment, for example `admin@tstg.bigtbx.com`.

### Step B: Identity Alignment

Update the WordPress/notification sender identity so the effective `From` name and `From` email match the authenticated domain and mailbox used for SMTP delivery.

In practice, that includes reviewing:

- WordPress General Settings
- LoginPress notification/email settings
- any mail-plugin forced sender overrides

### Step C: SPF/DKIM Alignment

Ensure the sending domain used for invite mail has correct SPF and DKIM alignment so downstream corporate systems do not treat the message as spoofed or unauthenticated.

### Step D: Re-test With Real Accounts

After SMTP and sender identity are aligned:

- resend the password reset/invite for Robert Scott
- resend the password reset/invite for Marco Parrotto
- confirm receipt outside the local server logs
- compare WordPress-level results with the repo-side logging added in `v3.1.3`

## 5. Plugin/Environment Follow-Up

Both LoginPress and LoginWP are active according to the current operational digest. That remains a possible secondary source of interference, especially if redirects or onboarding behavior occur before a newly created user is fully settled.

Recommended follow-up if mail alignment alone does not resolve the issue:

- audit LoginPress email behavior
- audit LoginWP redirect behavior for non-admin users
- inspect any headless or optimization layer touching login, registration, or notification hooks

## 6. Recommended Next Instruction

If handing this off to an operator or another coding agent, the next instruction should be:

> Apply authenticated SMTP configuration, verify the effective sender identity in WordPress/LoginPress settings, and then resend the password reset for Robert and Marco to confirm delivery.
