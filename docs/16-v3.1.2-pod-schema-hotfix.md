# v3.1.2 Release Notes: POD Schema Hotfix

## Overview

Version 3.1.2 is a targeted follow-up to the v3.1.1 carrier POD release. In staging, POD uploads failed with MySQL error `Unknown column 'pod_path' in 'field list'`, which showed that the live `sl_transactions` table had not actually picked up the `pod_path` column expected by the new upload flow.

## Root Cause

The v3.1.1 code correctly wrote carrier POD URLs into `pod_path`, but older environments were relying on the existing `dbDelta()` table sync to alter the already-created `sl_transactions` table. In the failing environment, that alteration did not take effect, leaving the code and schema out of sync.

In practical terms:

- The receiver UI uploaded the file successfully enough to reach the save step.
- `TransactionController::uploadPOD()` assigned the uploaded file URL to `$transaction->pod_path`.
- Eloquent generated an `UPDATE` against the prefixed `sl_transactions` table.
- MySQL rejected the statement because `pod_path` was missing from the live table.

## Fix

`TableManager` now performs an explicit runtime schema check for `sl_transactions.pod_path` and adds the column if it is missing.

Implementation details:

- Added a `TRANSACTIONS_TABLE` constant for the canonical transactions table name.
- Added `TRANSACTION_COLUMN_UPGRADES` to define guarded one-off column additions.
- Added `ensureTransactionColumnUpgrades()` to:
  - confirm the prefixed transactions table exists
  - check whether `pod_path` exists
  - run `ALTER TABLE ... ADD COLUMN` only when needed
  - log success or failure for visibility
- Called the upgrade guard from `TableManager::init()` immediately after `createTransactionsTable()`

## Why This Release Exists

This is fundamentally a schema mismatch hotfix. The failure was not caused by PDF-specific handling, receiver selection, or permissions. The upload request was reaching the database layer and dying because SQL could not find the destination column.

## Operational Notes

- Deploying v3.1.2 should allow older environments to self-heal on the next WordPress load, assuming the database user can alter the table.
- Staging verification after plugin upload and activation confirmed that the `pod_path` column was created successfully on the transactions table.
- If an environment is still blocked before the new code is deployed, the equivalent manual recovery SQL is:

```sql
ALTER TABLE `<prefix>sl_transactions`
ADD COLUMN `pod_path` VARCHAR(255) NULL AFTER `image_path`;
```

- After deployment, re-test:
  - upload POD
  - replace POD
  - delete POD
  - view/print POD

## Safety Check: Could App Usage Drop The Column?

No evidence in the plugin suggests that normal application use could remove `pod_path` after it has been created.

What the code does:

- `TableManager` runs `dbDelta()` for table definitions and a guarded `ALTER TABLE ... ADD COLUMN` only when `pod_path` is missing.
- `TransactionController::uploadPOD()` writes a URL string into `pod_path`.
- `TransactionController::deletePOD()` clears the field value by setting `pod_path` to `NULL`.

What the code does not do:

- no `DROP COLUMN`
- no `DROP TABLE`
- no destructive `ALTER TABLE` that removes or rewrites `pod_path`

So the realistic risk is not that users will somehow make the column disappear through normal uploads or deletes. The more plausible failure modes would be an external database change, a manual schema rollback, or a deployment of older plugin code over the newer version.
